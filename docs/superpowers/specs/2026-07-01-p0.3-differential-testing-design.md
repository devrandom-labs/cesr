# P0.3 — Differential testing vs keripy (design)

> Issue: [#27 · P0.3 Differential testing vs keripy](https://github.com/devrandom-labs/cesr/issues/27)
> Milestone: Phase 0 · Instruments
> keripy pin: `v2.0.0.dev5`
> Status: approved design, pre-implementation

## Problem

The `keripy-sync` watcher proves **code-table** parity: cesr defines the same code
points keripy does. It does **not** prove **behavioral** parity — that cesr and
keripy serialize and parse the *same bytes for the same value*. Differential
testing is the real interop guarantee and the strongest correctness guardrail for
the Phase 1 zero-copy / performance refactors that follow.

## Prior art in this repo (do not reinvent)

Two existing patterns solve the same shape and are reused here:

1. **keripy-generated, checked-in, hermetically-replayed vectors.** The committed
   Matter vectors (`src/core/matter/test_vectors*.rs`) carry the header
   *"Generated from KERIpy v2.0.0-dev5"* via a `scripts/generate_test_vectors.py`
   that was **never committed** (it arrived with the initial agency source copy in
   #2). So the repo already has keripy-derived, gate-replayed vectors — but only
   for Matter, with the generator lost and no live-diff loop.
2. **Hermetic-gate + nightly-side-workflow split.** The fuzzing work (#26) replays
   a checked-in corpus deterministically inside `nix flake check` and runs live
   fuzzing in a separate nightly workflow. `keripy-sync` likewise runs keripy in a
   weekly workflow, never in the gate, and takes a keripy checkout via `--keripy`
   (no network in-script), pinning `v2.0.0.dev5`.

P0.3 generalizes pattern (1) to Counter/Indexer/stream and revives the generator,
structured as the two tiers of pattern (2).

## Architecture — two-tier

### Tier 1 · Hermetic gate

Runs inside `nix flake check` via `cargo nextest` (integration tests under
`tests/` are already collected — **no `flake.nix` change required**).

- A checked-in, keripy-generated corpus of vectors for **Matter, Counter v1,
  Counter v2, Indexer, and a composed stream**.
- A Rust integration test (`tests/keripy_diff.rs`) replays each vector two ways:
  - **decode**: parse keripy's `qb64` and `qb2` → assert the reconstructed fields
    (code, raw, soft, index/ondex, count/version) equal the vector's fields.
  - **encode**: build the value from its structured fields → assert the emitted
    `qb64` and `qb2` equal keripy's exact bytes.
- Zero keripy dependency at gate time. This is the byte-exact behavioral oracle.

### Tier 2 · Nightly live-diff

A separate `.github/workflows/keripy-diff.yml`, non-hermetic — mirrors the existing
nightly deep-fuzz split.

- `scripts/keripy_diff_gen.py` (generalized revival of the lost generator),
  **stdlib + keripy only**, takes `--keripy <checkout>` like `sync.py`; the
  workflow performs the pinned checkout of `v2.0.0.dev5` (no network in-script).
- Input generation: a **boundary-size sweep** (`0, 1, MAX-1, MAX` per code table,
  per CLAUDE.md test rules) plus random fill, plus curated vectors harvested from
  keripy `tests/core/test_*.py` where practical.
- For each input, run keripy encode/decode → emit JSONL vectors → run the diff
  harness against the freshly generated corpus. Any disagreement fails the job.
  On new coverage it opens a PR to extend the committed corpus (as `keripy-sync`
  opens its report PR).

## Corpus format — JSONL data files

`tests/corpus/keripy/{matter,counter_v1,counter_v2,indexer,stream}.jsonl`, one
vector per line:

```json
{"kind":"matter","code":"D","raw":"<hex>","soft":"","qb64":"D...","qb2":"<hex>"}
```

Fields are per-kind (index/ondex for indexer; count/version for counter). Read by
the Rust test via `serde_json`, added to `[dev-dependencies]` so the diff test has
it regardless of crate feature selection.

**Why JSONL over the existing generated-`.rs` convention:** an append-growable
corpus diffs cleanly line-by-line, is language-neutral (Python writes, Rust reads),
and avoids ever-growing multi-thousand-line generated Rust files. The existing
Matter `TestVector` `.rs` files remain as the unit-level tier; this is the new
cross-implementation tier.

## Pinning

keripy **`v2.0.0.dev5`** — the single tag already used by `keripy-sync` and the
existing Matter vectors. One source of truth; bump deliberately.

## Error handling & acceptance

- Any decode-mismatch, encode-mismatch, or parse-failure on a keripy-valid vector
  is a hard test failure that prints the offending vector.
- A genuine cesr↔keripy disagreement is filed as a bug issue rather than silently
  tolerated (satisfies the AC "zero disagreements on the covered surface, or files
  bugs for any").
- Input-validation failures (malformed corpus line) are distinct from
  parity-mismatch failures — different assertion sites, different messages.

## Modules / boundaries

| Unit | Purpose | Depends on |
|------|---------|------------|
| `tests/keripy_diff.rs` | Hermetic replay harness; one fn per primitive kind | corpus files, `serde_json`, public encode/decode API |
| `tests/corpus/keripy/*.jsonl` | Frozen keripy-generated oracle vectors | — |
| `scripts/keripy_diff_gen.py` | Tier-2 generator (`--keripy` path; boundary + random) | a keripy checkout |
| `.github/workflows/keripy-diff.yml` | Nightly: pinned checkout → regen → test → PR | generator, keripy tag |

## Open items (resolved during implementation, flagged now)

1. **Encode-path normalization.** The encode surface is heterogeneous: `Indexer`
   and `Siger` expose `to_qb64()`/`to_qb2()` directly, but `Matter`'s encode path
   runs through `serder::primitives::to_qb64_string()` (behind the `serder`
   feature), and `Counter`'s public encode API is not in `counter/mod.rs`. Confirm
   the public encode entry point per primitive and select the feature set the diff
   test compiles under so all four primitives encode. `stream/binary.rs`
   (`qb64_to_qb2` / `qb2_to_qb64`) provides an independent transcoder cross-check.
2. **Composed-stream shape.** Define the exact multi-primitive stream both keripy
   and cesr round-trip (Counter framing + N Matters), and how the harness walks it.

## Non-goals

- No `pyo3` / in-process linking and no subprocess-per-input in the gate (both
  break `nix flake check` hermeticity — explicitly rejected).
- No change to the existing unit-level Matter `test_vectors*.rs`.
- No coverage beyond Matter / Counter v1+v2 / Indexer / one composed stream in this
  iteration.
